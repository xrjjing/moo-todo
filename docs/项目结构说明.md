# 牛牛待办项目结构说明

## 1. 文档用途

这份文档用于帮助新接手项目的人快速回答下面几类问题：

- 这个仓库的主要目录分别负责什么
- 前端页面、前端脚本、后端入口、服务层、数据层是怎么串起来的
- 某个功能出问题时，应该先看页面、JS、API 还是 Service
- 当前这轮“深度注释整理”主要覆盖了哪些核心文件


## 2. 建议阅读顺序

如果你是第一次看这个项目，建议按下面顺序阅读：

1. `main.py`
   - 先确认桌面应用是如何启动、如何把前端页面挂到 pywebview 窗口里的。
2. `web/index.html`
   - 先看页面骨架，知道主界面有哪些区域、有哪些弹窗和悬浮层。
3. `web/app.js`
   - 再看前端交互入口、初始化流程、视图切换、弹窗控制、API 调用。
4. `api.py`
   - 看前端通过 `window.pywebview.api` 能调用哪些后端方法。
5. `services/todo_service.py`
   - 看待办、分类、番茄钟、设置、统计、成就等核心业务是怎么落地的。
6. `services/db_manager.py`
   - 看 SQLite 表结构、通用 CRUD、设置项和 JSON 字段序列化。
7. `services/ai_manager.py` / `services/ai_providers.py`
   - 最后看 AI Provider 管理、会话管理、模型获取和实际 HTTP 调用。


## 3. 顶层目录与文件职责

### 3.1 顶层目录

- `web/`
  - 前端资源目录。
  - 包含主页面 `index.html`、样式 `styles.css`、前端脚本 `app.js`、图表库 `lib/`。
- `services/`
  - 后端业务逻辑目录。
  - 负责任务管理、SQLite 持久化、AI Provider 管理与聊天会话管理。
- `tests/`
  - 自动化测试目录。
  - 当前主要是 `test_todo_service.py`，偏向验证待办服务核心行为。
- `icons/`
  - 应用图标原始资源。
- `dist/`
  - 打包产物目录，通常不是阅读业务逻辑的入口。
- `docs/`
  - 项目文档目录。
  - 当前新增这份结构说明，专门承接“目录职责 + 调用链”说明，不把 README 堆得过重。

### 3.2 顶层核心文件

- `main.py`
  - 桌面应用启动入口。
  - 负责创建 `Api` 实例、定位 `web/index.html`、启动 pywebview 窗口。
- `api.py`
  - 前后端桥接层。
  - 前端通过 `window.pywebview.api.xxx()` 调用这里的方法。
- `README.md`
  - 面向使用者/贡献者的项目说明。
  - 适合看功能概览、运行方式、打包方式，不适合承载完整调用链细节。
- `build.py`
  - 打包辅助脚本。
- `pyproject.toml`
  - Python 项目元信息、依赖与 pytest 配置。
- `requirements.txt`
  - 基础运行依赖。


## 4. 前端结构

### 4.1 页面入口

- 页面文件：`web/index.html`
- 脚本文件：`web/app.js`
- 样式文件：`web/styles.css`

页面是一个典型的“**单页面桌面应用界面**”：

- `index.html` 负责定义静态骨架和所有容器
- `app.js` 负责初始化、绑定事件、请求后端、渲染动态内容
- `styles.css` 负责视觉表现，不参与数据流

### 4.2 页面骨架分层

`web/index.html` 可以从外到内分成几层：

1. **应用总容器**
   - `.app`
   - 承载侧边栏和主内容区。
2. **左侧导航/信息区**
   - `.sidebar`
   - 放 Logo、视图切换、统计摘要、筛选条件、分类/标签入口等。
3. **右侧主工作区**
   - `.content`
   - 放搜索框、顶部操作按钮、新建任务按钮、四种主视图容器。
4. **全局弹窗层**
   - 任务弹窗、分类弹窗、设置弹窗、总结弹窗、成就弹窗、AI 聊天弹窗、AI 设置弹窗等。
5. **全局悬浮层**
   - 番茄钟悬浮窗、便签悬浮窗、Toast。

### 4.3 前端脚本主要分块

`web/app.js` 已经按功能做了较清晰的分段，阅读时可以重点看：

- 状态管理
  - 保存任务列表、分类、当前视图、番茄钟状态、快捷键状态、AI 会话状态等。
- 初始化
  - `DOMContentLoaded` 后等待 `pywebview.api` 可用，再加载分类、标签、任务与各类 UI 初始化逻辑。
- 视图系统
  - 列表、看板、日历、四象限四类主视图切换与渲染。
- 任务弹窗 / 分类弹窗
  - 表单读写、编辑态回填、提交后刷新。
- 番茄钟 / 统计 / 成就
  - 与任务完成、统计图表、成就检查相关。
- AI 聊天 / AI 设置
  - Provider 配置、会话列表、聊天消息读写。


## 5. 后端结构

### 5.1 启动层

- `main.py`
  - 创建 `Api()`。
  - 通过 `webview.create_window(..., js_api=api)` 把桥接对象暴露给前端。
  - 前端页面加载后，通过 `window.pywebview.api` 调用 `api.py` 中的方法。

### 5.2 API / 桥接层

- `api.py`
  - 这是前端调用的直接入口层。
  - 主要作用不是承载复杂业务，而是：
    - 接收前端参数
    - 调用 `TodoService` 或 `AIManager`
    - 做基础异常包装
    - 把 dataclass 转成字典返回给前端

`Api` 里的方法按职责大致可分为：

- 任务相关
  - `add_task`、`get_tasks`、`update_task`、`delete_task`
- 子任务相关
  - `add_subtask`、`toggle_subtask`、`delete_subtask`
- 分类相关
  - `add_category`、`get_categories`
- 番茄钟相关
  - `start_pomodoro`、`complete_pomodoro`
- 统计/总结相关
  - `get_stats`、`get_tasks_by_date_range`、`get_pomodoro_daily_stats`
- 设置/主题/缩放/快捷键
  - `get_settings`、`update_settings`、`save_theme`、`save_zoom`、`save_shortcuts`
- 数据导入导出
  - `export_db`、`import_db`、`get_data_stats`
- 成就
  - `get_achievements`、`check_achievements`
- AI Provider / AI Chat
  - `get_ai_providers`、`save_ai_provider`、`create_chat_session`、`send_chat_message` 等

### 5.3 业务服务层

- `services/todo_service.py`
  - 核心业务服务。
  - 负责把数据库里的任务、分类、番茄钟、设置加载到内存缓存中，并提供统一业务操作入口。
- `services/ai_manager.py`
  - AI 统一管理器。
  - 负责 Provider 配置加载、活跃 Provider 切换、聊天会话与消息管理、模型列表获取、连接测试。
- `services/ai_providers.py`
  - AI Provider 适配层。
  - 负责把统一消息格式转换为不同厂商需要的 HTTP 请求。

### 5.4 数据持久化层

- `services/db_manager.py`
  - SQLite 数据访问层。
  - 负责：
    - 初始化表结构
    - 提供通用 CRUD
    - 处理设置项、活跃配置
    - 处理 JSON 字段序列化/反序列化


## 6. 关键数据流与调用关系

## 6.1 总体调用链

整体链路是：

`页面结构(index.html) -> 前端交互(app.js) -> pywebview API(api.py) -> 业务服务(TodoService / AIManager) -> SQLite(DatabaseManager)`

其中：

- `index.html` 决定“页面上有哪些坑位和按钮”
- `app.js` 决定“什么时候触发、调哪个 API、把结果渲染到哪里”
- `api.py` 决定“前端能调用哪些后端能力”
- `TodoService` / `AIManager` 决定“具体业务怎么做”
- `DatabaseManager` 决定“数据怎么落盘和取回”

### 6.2 任务列表主链路

1. 页面加载完成后，`app.js` 的初始化流程会执行 `loadTasks()`
2. `loadTasks()` 从筛选器和搜索框读取条件
3. 调用 `pywebview.api.get_tasks(...)`
4. `api.py` 把请求转给 `TodoService.get_tasks(...)`
5. `TodoService` 从内存缓存中过滤任务并返回结果
6. `app.js` 根据 `state.currentView` 调用：
   - `renderListView()`
   - `renderKanbanView()`
   - `renderCalendarView()`
   - `renderQuadrantView()`
7. 结果被渲染到 `index.html` 中对应的视图容器里

### 6.3 新建/编辑任务链路

1. 用户点击“+ 新任务”或任务卡片
2. `app.js` 打开任务弹窗并收集表单字段
3. 调用：
   - 新建：`pywebview.api.add_task(...)`
   - 编辑：`pywebview.api.update_task(...)`
4. `api.py` 转给 `TodoService`
5. `TodoService` 更新内存任务对象，并同步写入数据库
6. 前端重新 `loadTasks()`，刷新主视图和统计摘要

### 6.4 子任务链路

1. 用户在任务编辑弹窗中增删改子任务
2. `app.js` 调用 `add_subtask` / `toggle_subtask` / `delete_subtask`
3. `api.py` 转给 `TodoService`
4. `TodoService` 同时更新：
   - 任务对象中的 `subtasks`
   - `subtasks` 表中的对应记录

### 6.5 番茄钟链路

1. 用户在任务卡片或相关操作里启动番茄钟
2. `app.js` 调用 `pywebview.api.start_pomodoro(taskId, duration)`
3. `TodoService.start_pomodoro(...)` 创建番茄记录
4. 前端显示 `#pomodoro-widget` 悬浮窗并本地计时
5. 计时结束后，前端再调用 `complete_pomodoro`
6. 后端把记录标记为完成，并更新任务番茄次数/统计数据

### 6.6 工作总结 / 统计链路

1. 用户打开统计或总结弹窗
2. `app.js` 并行请求：
   - `get_stats(...)`
   - `get_tasks_by_date_range(...)`
   - `get_pomodoro_daily_stats(...)`
   - `get_pomodoro_heatmap(...)`
   - `get_category_pomodoro_stats(...)`
3. `TodoService` 负责汇总内存数据或数据库数据
4. 前端把结果渲染到统计卡片、趋势图、热力图、任务清单等区域

### 6.7 AI 聊天链路

1. 用户打开 AI 助手弹窗
2. `app.js` 先加载 Provider 列表和会话列表
3. 发送消息时调用 `pywebview.api.send_chat_message(...)`
4. `api.py` 会：
   - 先把用户消息写入聊天会话
   - 再调用 `AIManager.chat_with_history(...)`
5. `AIManager` 根据当前 Provider 找到具体实现类
6. `ai_providers.py` 中的 Provider 负责发起 HTTP 请求
7. 返回结果后：
   - 后端把 AI 回复写回 `chat_messages`
   - 前端刷新会话消息区


## 7. 关键文件的排查建议

### 7.1 页面显示不对

优先看：

1. `web/index.html`
   - 是否少了容器、id、按钮入口
2. `web/styles.css`
   - 是否是样式问题
3. `web/app.js`
   - 对应区域是否没有渲染、是否没有给容器赋值

### 7.2 按钮点了没反应

优先看：

1. `index.html` 上按钮的 `onclick` 或输入事件
2. `app.js` 对应函数是否存在
3. 该函数里有没有调用 `pywebview.api.xxx`
4. `api.py` 里是否有同名方法

### 7.3 数据没保存 / 重启后丢失

优先看：

1. `api.py` 是否真正调用到了 `TodoService`
2. `TodoService` 是否调用了 `db.insert` / `db.update`
3. `DatabaseManager` 的表结构和写入逻辑是否正常
4. 数据库文件位置是否仍是 `~/.todo_app/moo_todo.db`

### 7.4 AI 对话异常

优先看：

1. `web/app.js` 的 AI 弹窗与消息发送逻辑
2. `api.py` 的 AI Provider / Chat 方法
3. `services/ai_manager.py`
   - 当前活跃 Provider 是否正确
   - 会话和消息是否入库
4. `services/ai_providers.py`
   - 请求头、URL、模型名、错误处理是否命中


## 8. 当前这轮工程化整理覆盖范围

这轮整理聚焦三类产物：

1. **项目结构文档**
   - 本文件：`docs/项目结构说明.md`
2. **前端深度注释**
   - 重点是 `web/index.html`
3. **后端深度注释**
   - 重点是 `main.py`、`api.py`、`services/*.py`

整理原则是：

- 不改业务逻辑
- 只补结构化中文注释
- 注释尽量贴近真实上下游，不写没核实的推断
- 帮助后续排查时快速定位“页面层 / 脚本层 / API 层 / Service 层 / 数据层”
