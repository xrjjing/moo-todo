# 项目结构与调用链说明

## 1. 文档目标

这份文档用于帮助首次接手 `moo-todo` 的维护者快速建立整体认知，重点回答：

- 这个项目的主要目录和文件分别负责什么
- 前端页面、前端脚本、后端入口、API 桥接层、Service 层、数据层如何衔接
- 常见功能从“页面操作”到“后端落库”的主要调用链是什么
- 排查问题时应该先看哪一层、哪几个关键文件

---

## 2. 推荐阅读顺序

建议按下面顺序理解项目：

1. `main.py`：看桌面应用是如何启动的
2. `web/index.html`：看页面骨架、弹窗和各工作区长什么样
3. `web/app.js`：看页面交互、渲染函数、`pywebview.api` 调用入口
4. `api.py`：看前端能调用哪些后端方法，以及这些方法分组
5. `services/todo_service.py`：看任务、番茄钟、统计、设置、成就等核心业务
6. `services/db_manager.py`：看 SQLite 表结构和通用 CRUD
7. `services/ai_manager.py` / `services/ai_providers.py`：看 AI Provider 配置、会话、消息和模型调用

## 4. 前后端分层关系

## 4.1 总体架构

这个项目不是前后端分离 Web 服务，而是：

- **Python** 负责本地桌面应用、业务逻辑、数据库读写、AI 调用
- **pywebview** 负责把前端页面嵌入桌面窗口
- **HTML/CSS/原生 JS** 负责界面和交互
- **`window.pywebview.api`** 负责前端 JS 与 Python 的桥接

可把它理解为：

```text
用户点击页面
  -> web/index.html 中的按钮 / 输入框 / 弹窗
  -> web/app.js 中的事件函数与渲染函数
  -> window.pywebview.api.*
  -> api.py 中的 Api 方法
  -> services/*.py 中的业务服务
  -> DatabaseManager / AI Provider
  -> 结果回到 app.js
  -> 更新 DOM
```

## 4.2 各层职责边界

### 1）桌面入口层：`main.py`

- 创建 `Api()` 实例
- 定位 `web/index.html`
- 用 `webview.create_window(...)` 创建桌面窗口
- 通过 `js_api=api` 把 Python 方法暴露给前端 JS

### 2）页面结构层：`web/index.html`

- 提供页面骨架和所有静态容器
- 包含：
  - 侧边栏
  - 顶部栏
  - 过滤区
  - 四种主视图容器（列表 / 看板 / 日历 / 四象限）
  - 新建/编辑任务弹窗
  - 分类弹窗
  - 工作总结弹窗
  - 设置弹窗
  - 快捷键弹窗
  - 番茄钟悬浮窗
  - 便签悬浮窗
  - 专注统计弹窗
  - 成就弹窗
  - AI 聊天弹窗
  - AI 设置弹窗
- 自己几乎不承载业务逻辑，主要提供 **DOM 锚点**

### 3）前端交互层：`web/app.js`

- 统一维护前端状态，如 `state.tasks`、当前视图、番茄钟状态、AI 会话状态
- 页面初始化时等待 `window.pywebview.api` 可用，再批量加载分类、标签、任务
- 负责：
  - DOM 事件绑定
  - 视图切换
  - 列表 / 看板 / 日历 / 四象限渲染
  - 弹窗开关
  - 表单收集
  - 调用 `pywebview.api.*`
  - 拿到结果后回填页面

### 4）桥接层：`api.py`

- 这是前端 JS 唯一直接接触的 Python 层
- 主要作用：
  - 对外暴露统一方法名
  - 使用错误处理装饰器统一返回错误结构
  - 把任务、统计、设置等调用转发给 `TodoService`
  - 把 AI 相关调用转发给 `AIManager`
- `AIManager` 采用 **懒加载**，只有前端真正打开 AI 功能时才初始化

### 5）核心业务层：`services/todo_service.py`

- 项目最重的业务文件
- 内部同时承担：
  - 任务 CRUD
  - 子任务 CRUD
  - 重复任务生成
  - 分类管理
  - 番茄钟记录
  - 专注统计
  - 工作总结统计
  - 主题 / 缩放 / 快捷键设置
  - 数据导入导出
  - 成就计算与解锁
- 这个文件还维护了一层 **内存缓存**：启动时把数据库数据加载到内存对象中，后续业务操作再同步回数据库

### 6）数据层：`services/db_manager.py`

- 负责 SQLite 初始化和所有表结构创建
- 提供通用 CRUD：
  - `insert`
  - `update`
  - `delete`
  - `get_by_id`
  - `get_all`
- 同时处理设置项、活跃配置、JSON 字段序列化/反序列化

### 7）AI 管理层：`services/ai_manager.py`

- 负责从数据库加载 Provider 配置
- 负责切换活跃 Provider
- 负责测试连接、获取模型列表
- 负责聊天会话、消息历史、请求统计
- 对具体 AI 厂商差异做一层统一封装

### 8）AI Provider 实现层：`services/ai_providers.py`

- `AIProvider`：抽象基类
- `OpenAIProvider`：对接 OpenAI Chat Completions
- `ClaudeProvider`：对接 Anthropic Messages API
- `ThirdPartyProvider`：对接 OpenAI 兼容接口

## 5. 前端页面结构总览

### 5.1 页面主骨架

## 6. 核心文件细分说明

## 6.1 前端相关

### `web/index.html`

定位：

- 唯一页面模板
- 所有视图和弹窗都集中定义在这里

重点区域：

- 左侧 `sidebar`
  - Logo 区
  - 视图切换按钮
  - 分类列表 `categories-list`
  - 今日统计卡片
  - 主题切换菜单
- 右侧 `content`
  - 搜索框 `search-input`
  - 顶部按钮区（设置 / 统计 / 总结 / 成就 / 便签 / AI / 新任务）
  - 过滤器（状态 / 优先级 / 分类 / 标签）
  - 四种视图容器
- 底部全局浮层
  - 任务编辑相关
  - 设置相关
  - AI 相关
  - 番茄钟 / 便签 / toast

排查建议：

- **看不到某个区域**：先查 `index.html` 是否有对应 DOM 容器
- **能看到壳子但没数据**：再查 `app.js` 对应渲染函数有没有往该容器写入内容

### `web/app.js`

定位：

- 单文件前端控制器
- 几乎所有前端行为都集中在这里

主要模块：

- 初始化：等待 `pywebview.api`、初始化主题/缩放/视图/数据
- 任务模块：加载、渲染、搜索、增删改、状态切换
- 多视图模块：列表、看板、日历、四象限
- 子任务模块
- 重复任务模块
- 分类模块
- 番茄钟模块
- 总结与统计模块
- 设置与快捷键模块
- 便签模块
- AI 会话 / 消息 / Provider 配置模块

排查建议：

- **点击没反应**：先搜 HTML 上的 `onclick` / DOM 事件，再定位到 `app.js`
- **接口调用异常**：查 `pywebview.api.xxx` 对应的 `api.py` 方法
- **页面数据不更新**：优先检查对应 `render*` / `load*` 函数

### `web/styles.css`

- 纯样式文件
- 影响布局、颜色、弹窗、卡片、图表容器、主题表现
- 本轮主要做结构说明，不涉及逻辑链路

## 6.2 后端相关

### `main.py`

- 程序入口
- 负责把 `Api` 对象挂给前端 JS
- 相当于整个项目的“壳”

### `api.py`

- 前端 JS 到 Python 的第一站
- 方法分组比较清晰：
  - Task API
  - Subtask API
  - Recurring API
  - Category API
  - Pomodoro API
  - Stats API
  - Settings API
  - Shortcuts API
  - Import/Export API
  - Achievement API
  - AI Provider API
  - AI Chat API

排查建议：

- **前端说某 API 不存在**：先看这里有没有同名方法
- **前端收到错误结构**：先看装饰器 `api_error_handler` / `async_api_handler`

### `services/todo_service.py`

- 项目核心业务中心
- 同时包含数据模型和业务操作
- 关键点：
  - 启动时 `_load_all()` 把数据库数据装入内存
  - 对任务、分类、番茄钟、设置等操作时，再同步写回数据库
  - 成就、统计、重复任务等也都集中在这里

排查建议：

- **任务数据不对**：从 `get_tasks` / `add_task` / `update_task` / `delete_task` 开始查
- **番茄钟统计不对**：从 `start_pomodoro` / `complete_pomodoro` / 统计函数开始查
- **工作总结不对**：从 `get_stats`、`get_tasks_by_date_range` 查
- **成就不对**：看 `_calculate_progress`、`_calculate_streak`、`check_achievements`

### `services/db_manager.py`

- 数据库底座
- 表结构全在 `_init_database()` 中
- 所有 JSON 字段的入库/出库转换也在这里

排查建议：

- **数据库表缺失/字段异常**：看 `_init_database`
- **读出来是字符串不是对象**：看 `_deserialize_json_fields`
- **写入失败**：看 `insert` / `update` / `execute_update`

### `services/ai_manager.py`

- AI 功能总控层
- 管理 Provider 生命周期、会话、消息和统计

排查建议：

- **AI Provider 列表不对**：看 `_load_config`、`get_available_providers`
- **测试连接/模型拉取失败**：看 `test_connection`、`fetch_models`
- **聊天历史不对**：看 `create_session`、`add_message`、`get_messages`

### `services/ai_providers.py`

- 不同 AI 厂商的真实 HTTP 调用都在这里

排查建议：

- **OpenAI 请求失败**：看 `OpenAIProvider.chat`
- **Claude 请求失败**：看 `ClaudeProvider.chat`
- **兼容接口异常**：看 `ThirdPartyProvider`

## 7. 重要调用链

下面是当前项目最值得优先掌握的调用链。

## 7.1 应用启动链路

```text
启动 main.py
  -> main()
  -> Api()
  -> webview.create_window(..., js_api=api)
  -> 加载 web/index.html
  -> web/app.js 在 DOMContentLoaded 后执行
  -> waitForApi() 等待 window.pywebview.api 就绪
  -> 初始化主题 / 缩放 / 视图 / 分类 / 标签 / 任务
```

## 7.2 任务列表加载链路

```text
页面初始化 / 搜索 / 过滤器变化
  -> app.js: loadTasks()
  -> pywebview.api.get_tasks(...)
  -> api.py: Api.get_tasks(...)
  -> todo_service.py: TodoService.get_tasks(...)
  -> 从内存任务列表筛选
  -> 返回前端
  -> app.js: renderCurrentView()
  -> 渲染列表 / 看板 / 日历 / 象限中的当前视图
```

## 7.3 新建任务链路

```text
点击“+ 新任务”
  -> app.js: showTaskModal()
  -> 填写弹窗表单
  -> app.js: saveTask()
  -> pywebview.api.add_task(...)
  -> api.py: Api.add_task(...)
  -> todo_service.py: TodoService.add_task(...)
  -> db_manager.py: insert('tasks', ...)
  -> 返回前端
  -> app.js: closeModal('task-modal')
  -> app.js: loadTasks()
```

## 7.4 编辑任务链路

```text
点击任务卡片 / 看板卡片
  -> app.js: showEditTaskModal(taskId)
  -> 弹窗回填任务数据
  -> app.js: saveTask()
  -> pywebview.api.update_task(taskId, {...})
  -> api.py: Api.update_task(...)
  -> todo_service.py: TodoService.update_task(...)
  -> db_manager.py: update('tasks', ...)
  -> 前端重新 loadTasks()
```

## 7.5 任务完成状态切换链路

```text
点击任务勾选框
  -> app.js: toggleTaskStatus(taskId)
  -> pywebview.api.update_task_status(taskId, status)
  -> api.py: Api.update_task_status(...)
  -> todo_service.py: update_task_status()
  -> todo_service.py: update_task(..., status=...)
  -> 若完成则补 completed_at
  -> db_manager.py: update('tasks', ...)
  -> 前端刷新任务与统计
```

## 7.6 子任务链路

```text
任务弹窗中新增 / 勾选 / 删除子任务
  -> app.js: addSubtask / toggleSubtask / deleteSubtask
  -> pywebview.api.add_subtask / toggle_subtask / delete_subtask
  -> api.py 对应方法
  -> todo_service.py 子任务 CRUD
  -> db_manager.py: subtasks 表增删改
  -> 前端刷新子任务列表与进度
```

## 7.7 番茄钟链路

```text
点击“开始专注”或任务相关入口
  -> app.js: startPomodoro(taskId)
  -> pywebview.api.start_pomodoro(taskId, duration)
  -> api.py: Api.start_pomodoro(...)
  -> todo_service.py: start_pomodoro(...)
  -> db_manager.py: insert('pomodoros', ...)
  -> 前端显示 pomodoro-widget 并本地倒计时

倒计时结束
  -> app.js: completePomodoro()
  -> pywebview.api.complete_pomodoro(pomodoroRecordId)
  -> todo_service.py: complete_pomodoro(...)
  -> db_manager.py: update('pomodoros', ...)
  -> app.js: loadTasks() / updateStats() / checkAndShowAchievements()
```

## 7.8 工作总结与统计链路

```text
打开“工作总结”弹窗
  -> app.js: showSummaryModal() / loadSummary()
  -> pywebview.api.get_stats(startDate, endDate)
  -> todo_service.py: get_stats(...)

同时
  -> pywebview.api.get_tasks_by_date_range(startDate, endDate)
  -> todo_service.py: get_tasks_by_date_range(...)

前端汇总
  -> 更新统计卡片
  -> 渲染任务列表
  -> 生成总结文本
```

## 7.9 专注统计图表链路

```text
打开“专注统计”弹窗
  -> app.js: showStatsModal()
  -> app.js: loadStatsData()
  -> Promise.all 并行调用：
       get_pomodoro_daily_stats(30)
       get_pomodoro_heatmap(当前年份)
       get_category_pomodoro_stats()
  -> api.py -> todo_service.py 统计函数
  -> 前端 renderTrendChart / renderHeatmap / renderCategoryStats
```

## 7.10 AI 配置与聊天链路

```text
打开 AI 弹窗
  -> app.js: showAIChatModal()
  -> pywebview.api.get_ai_providers()
  -> pywebview.api.get_chat_sessions()
  -> api.py -> AIManager

发送消息
  -> app.js: sendAIMessage()
  -> 若无会话则先 create_chat_session()
  -> pywebview.api.send_chat_message(sessionId, content)
  -> api.py: send_chat_message(...)
  -> AIManager.add_message(保存用户消息)
  -> AIManager.chat_with_history(...)
  -> ai_providers.py 中具体 Provider 发 HTTP 请求
  -> AIManager.add_message(保存 AI 回复)
  -> 返回前端渲染聊天区
```

## 7.11 数据导入导出链路

```text
设置弹窗中导出 / 导入数据库
  -> app.js: exportData() / handleImportFile()
  -> pywebview.api.get_db_path() / export_db() / import_db()
  -> api.py -> todo_service.py
  -> todo_service.py 调用 shutil 复制数据库文件
  -> 导入后重新创建 DatabaseManager 并 _load_all()
  -> 前端 reload 页面
```

## 8. 当前结构特点与维护建议

## 8.1 当前结构特点

- 前端是 **单页面 + 单脚本主控**
- 后端是 **单 API 桥接 + 单核心业务服务**
- 数据存储是 **本地 SQLite**
- AI 能力是 **在既有待办应用之上挂载的扩展模块**

这意味着：

- 上手速度快
- 调用链相对直接
- 但 `web/app.js` 与 `services/todo_service.py` 都比较“大”，后续维护时要特别依赖注释和模块分区

## 8.2 排查问题的推荐顺序

### 界面显示问题

1. 先看 `web/index.html` 有没有这个区域
2. 再看 `web/app.js` 是否有对应渲染函数
3. 最后看 `styles.css` 是否把它隐藏了

### 点击无反应 / 数据没保存

1. 查 HTML 上绑定的事件
2. 查 `web/app.js` 对应函数
3. 查 `pywebview.api.xxx`
4. 查 `api.py` 是否存在同名方法
5. 查 `TodoService` / `AIManager` 是否真正落库或返回结果

### 数据异常 / 统计不准

1. 先看 `TodoService` 的业务计算
2. 再看 `db_manager.py` 的读写和 JSON 序列化
3. 若只影响 AI，再进入 `AIManager` / `ai_providers.py`
